## 2024-05-24 - [HIGH] Fix weak hashing algorithms
**Vulnerability:** MD5 and SHA-1 algorithms were being used for cryptographic hashes (e.g. `hashlib.md5` and `hashlib.sha1`) in several modules (`functions/cache.py`, `functions/telegram_bot.py`, `functions/user_storage.py`).
**Learning:** Usage of insecure algorithms like MD5 and SHA1 poses a security risk due to known vulnerabilities and collision weaknesses. Even when used for non-critical identifiers, it is a bad practice and violates security policies.
**Prevention:** Establish and strictly enforce a coding standard that mandates the use of modern, secure hash functions (such as SHA-256) for all hashing purposes. Here, we implemented a centralized `stable_hash` utility (using SHA-256) to ensure consistent and secure hashing across the codebase.

## 2024-05-25 - [HIGH] Server-Side Request Forgery (SSRF) risk in AI summarize endpoint
**Vulnerability:** The `summarize_url_callback` functionality fetched arbitrary user-provided URLs using `httpx.AsyncClient` while following redirects automatically (`follow_redirects=True`). This allowed a malicious user to supply an external URL that redirects to an internal server/service or private network address, leading to a Server-Side Request Forgery (SSRF) vulnerability.
**Learning:** Automatically following redirects with HTTP clients circumvents initial validation logic. Attackers can bypass naive URL checks by providing a benign URL that redirects to a sensitive internal resource.
**Prevention:** Always disable automatic redirect following (`follow_redirects=False`) for sensitive network requests that fetch user-provided URLs. Implement manual redirect handling up to a set limit, and validate every individual redirect hop against a robust safe URL check (e.g., rejecting private, local, and loopback IPs) before fetching.
