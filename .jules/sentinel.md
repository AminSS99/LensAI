## 2024-05-24 - [HIGH] Fix weak hashing algorithms
**Vulnerability:** MD5 and SHA-1 algorithms were being used for cryptographic hashes (e.g. `hashlib.md5` and `hashlib.sha1`) in several modules (`functions/cache.py`, `functions/telegram_bot.py`, `functions/user_storage.py`).
**Learning:** Usage of insecure algorithms like MD5 and SHA1 poses a security risk due to known vulnerabilities and collision weaknesses. Even when used for non-critical identifiers, it is a bad practice and violates security policies.
**Prevention:** Establish and strictly enforce a coding standard that mandates the use of modern, secure hash functions (such as SHA-256) for all hashing purposes. Here, we implemented a centralized `stable_hash` utility (using SHA-256) to ensure consistent and secure hashing across the codebase.

## 2024-05-24 - [HIGH] Fix Server-Side Request Forgery (SSRF) in URL summarization
**Vulnerability:** The application was fetching arbitrary user-provided URLs using `httpx.AsyncClient` with `follow_redirects=True` inside `functions/telegram_bot.py`. It didn't validate if the target URL (or any redirect destinations) resolved to private IP ranges, local networks, or the loopback interface, allowing potential SSRF.
**Learning:** Even simple proxying or fetching capabilities can be weaponized to map internal networks or access internal services (like AWS metadata, Redis, local APIs). Using automatic redirect following exacerbates the risk as an attacker can supply an external URL that redirects to an internal IP.
**Prevention:** Implemented an asynchronous `is_safe_url` validation utility that resolves hostnames and explicitly blocks loopback, private, unspecified, multicast, and reserved IP ranges. Integrated this check into a manual redirect-following loop in `summarize_url_callback`, disabling `follow_redirects` and validating every hop.
