## 2024-05-24 - [HIGH] Fix weak hashing algorithms
**Vulnerability:** MD5 and SHA-1 algorithms were being used for cryptographic hashes (e.g. `hashlib.md5` and `hashlib.sha1`) in several modules (`functions/cache.py`, `functions/telegram_bot.py`, `functions/user_storage.py`).
**Learning:** Usage of insecure algorithms like MD5 and SHA1 poses a security risk due to known vulnerabilities and collision weaknesses. Even when used for non-critical identifiers, it is a bad practice and violates security policies.
**Prevention:** Establish and strictly enforce a coding standard that mandates the use of modern, secure hash functions (such as SHA-256) for all hashing purposes. Here, we implemented a centralized `stable_hash` utility (using SHA-256) to ensure consistent and secure hashing across the codebase.
## 2026-03-31 - [SSRF Protection in summarize_url_callback]
**Vulnerability:** The application was fetching arbitrary user-provided URLs in `summarize_url_callback` using `httpx.AsyncClient` without validating the destination. This allowed Server-Side Request Forgery (SSRF), where an attacker could make the server request internal, loopback, or private IPs (e.g., cloud metadata endpoints like 169.254.169.254).
**Learning:** Naive SSRF protection is often bypassed via HTTP redirects. An attacker can provide a safe-looking external URL that redirects to an internal IP.
**Prevention:** To prevent SSRF securely, URLs must be validated by asynchronously resolving their hostname and checking the resulting IP addresses against reserved/private ranges using the `ipaddress` module. Furthermore, automatic redirects must be disabled (`follow_redirects=False`) and handled manually, ensuring that every URL in the redirect chain is independently validated before being fetched.
