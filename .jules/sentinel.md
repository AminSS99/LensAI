## 2024-05-24 - [HIGH] Fix weak hashing algorithms
**Vulnerability:** MD5 and SHA-1 algorithms were being used for cryptographic hashes (e.g. `hashlib.md5` and `hashlib.sha1`) in several modules (`functions/cache.py`, `functions/telegram_bot.py`, `functions/user_storage.py`).
**Learning:** Usage of insecure algorithms like MD5 and SHA1 poses a security risk due to known vulnerabilities and collision weaknesses. Even when used for non-critical identifiers, it is a bad practice and violates security policies.
**Prevention:** Establish and strictly enforce a coding standard that mandates the use of modern, secure hash functions (such as SHA-256) for all hashing purposes. Here, we implemented a centralized `stable_hash` utility (using SHA-256) to ensure consistent and secure hashing across the codebase.
## 2024-05-24 - [CRITICAL] Fix Server-Side Request Forgery (SSRF) vulnerability in URL summary fetching
**Vulnerability:** The `summarize_url_callback` handler in `functions/telegram_bot.py` was fetching content for user-provided URLs using `httpx.AsyncClient` with `follow_redirects=True` and no prior validation. This allowed an attacker to supply URLs pointing to internal services (like `localhost` or cloud metadata endpoints at `169.254.169.254`) and potentially extract sensitive data.
**Learning:** Never trust user-provided URLs, especially when fetching them server-side. Enabling automatic redirects in the HTTP client bypasses initial URL validation, as a safe initial URL could redirect to an internal endpoint.
**Prevention:** We created a centralized asynchronous `is_safe_url` utility in `functions/security_utils.py` that resolves the host IP and strictly filters out loopback, private, multicast, and other reserved IP addresses. In the fetch handler, we disabled automatic redirects (`follow_redirects=False`) and implemented manual redirect handling up to 5 hops, ensuring that every individual URL hop is independently verified by `await is_safe_url(url)` before the request is made.
